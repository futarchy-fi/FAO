# Rubric — Topic 5: Holistic Best Patterns (SC + Interfaces, Security + Maintainability)

> *Purpose: stateless Codex evaluator scores how coherent `futarchy-fi/FAO` is as
> an aggregate product — the contracts, the testnet site, the operator daemons,
> the docs, and the supply chain — and whether the **seams** between them are
> clean. This is the **meta rubric** for the CAO loop: it ingests the scores
> emitted by topics 1–4 and applies dependency caps before scoring its own
> dimensions. Anchors at 0 / 3 / 5 / 7 / 9. Aggregate is `min(dimension)` (the
> worst seam is the gate), with the arithmetic mean reported alongside. Pass
> threshold: **min ≥ 8.0**.*

Repo root: `/home/kelvin/repos/futarchy-fi/FAO`. Active branch at evaluation:
`arbitration/onchain-futarchy-v0`. Companion research:
`audit/research/topic-5-holistic-architecture.md`.

## 0. Scope and inputs

**In scope.** Cross-layer concerns: SC ↔ UI seam, deploy-artefact propagation,
admin / multisig / time-lock posture, deprecation hygiene (v2/v3/v4/v5),
operator daemons + runbooks, docs ↔ code freshness, supply-chain pins.

**Out of scope (other rubrics own these).**
- Web3 UX detail per page → Topic 1.
- Per-flow interface tests → Topic 2.
- Spec / invariant formalisation → Topic 3.
- Per-contract test infrastructure → Topic 4.
- LLM-maintained wiki → Topic 6.

**Required inputs.** The evaluator MUST read the most recent
`audit/evaluations/topic-{1,2,3,4}-*.json` (or `.md`) and extract their scores
before scoring this topic. The caps in §2 are computed from those scores. If a
prior evaluation is missing, treat that topic's score as the **median of its
anchors (5.0)** and document the assumption in the output.

**Output schema (per dimension).**

```
{ dimension, raw_score: 0–10, cap_applied: 0–10|null, final_score: min(raw,cap),
  evidence: [{file_or_command, line_or_pattern, note}], next_step: "..." }
```

Aggregate output:

```
TOPIC-5 SCORE (min): <x> / 10.0      ← gate
TOPIC-5 mean       : <x> / 10.0      ← informational
PASS: yes if min ≥ 8.0 else no
DEPENDENCY CAPS APPLIED:
  D2 capped by topic-3 (= <s>) → cap <c>
  D2 capped by topic-4 (= <s>) → cap <c>
  D5 capped by topic-1 (= <s>) → cap <c>
  D3 capped by topic-2 (= <s>) → cap <c>
```

## 1. Dimensions and anchors

Six dimensions. Each is 0–10; raw score is picked from the anchors below,
interpolated at even numbers. The **final** score for each dimension is
`min(raw_score, dependency_cap)` per §2.

### D1 — Architectural coupling discipline (SC ↔ UI ↔ deploy-artefact)

*Does the repo treat the ABI + deployment manifest as the single source of
truth for every consumer (UI, docs, daemon, Etherscan), or are addresses
hand-copied across surfaces?*

| Score | Anchor |
|------|--------|
| **0** | Addresses retyped into UI / docs / daemon as plain string literals; no manifest; ABI fragments copy-pasted into JS with subtly diverging signatures. |
| **3** | A `broadcast/` directory exists from `forge script`, but nothing post-processes it; UI hardcodes `const REGISTRY_ADDR = '0x...'` in ≥ 1 JS file; docs page lists a *different* address from the JS for the same logical role. |
| **5** | A markdown manifest (`docs/<network>-deployment-v0.md`) exists and is kept by hand; UI imports it at least conceptually but still has ≥ 1 inline hardcoded fallback; ABI fragments are hand-written but consistent across files. |
| **7** | A machine-readable manifest (`deployments/<chainId>.json`) is generated by the deploy script post-step; UI loads it at runtime or build time; ABI is generated (wagmi-cli / TypeChain / abitype); UI has no inline hardcoded primary address. |
| **9** | All of (7), plus docs site and operator daemon read the **same** manifest file; CI fails if the manifest diverges from `broadcast/run-latest.json`; address-set diff (§3) returns the empty set across `site*/`, `docs/`, `script/agents/`. |

Hard floor: if any one UI file imports an address that conflicts with any
docs file's address for the same logical role, raw score ≤ 3.

### D2 — Security posture (admin keys, immutability, timelocks, multisig)

*Are admin powers held by a Safe + Timelock, are math + state immutable, are
roles separated, are timelock delays sane?*

| Score | Anchor |
|------|--------|
| **0** | A single EOA holds every admin role; no Timelock anywhere; no multisig; pause / role-grant is instant. |
| **3** | OZ `AccessControl` is used inside contracts, but `DEFAULT_ADMIN_ROLE` is an EOA in production; no Timelock; testnet-only excuse is undocumented. |
| **5** | Roles are separated (`MINTER_ROLE`, `PAUSER_ROLE`, etc.); `DEFAULT_ADMIN_ROLE` lives on a Safe multisig, but **no Timelock** sits between Safe and contract; testnet-only single-key configuration is documented as such in `docs/`. |
| **7** | Production deploys use Safe → `TimelockController` → contract; delay ≥ 24h for parameter tweaks, 48h for role changes; immutable math libraries; a public `/governance` page lists pending Timelock actions with eta + decoded calldata. |
| **9** | All of (7), plus *every* admin address shown on Etherscan resolves to a Safe; testnet config and mainnet config are switched by env-var, not code edit; spec invariants (Topic 3) cover monotonicity of `accepted` / `resolved` / `nextProposalId`; Topic-4 invariant + fuzz tests prove the admin-role separations. |

Dependency caps:
- If `topic-3` < 5.0 → D2 capped at **6** (no spec ⇒ no contract-of-contract for upgrades).
- If `topic-4` < 5.0 → D2 capped at **6** (no fuzz / invariants ⇒ no defence-in-depth proof).
- If `topic-3` < 3.0 OR `topic-4` < 3.0 → D2 capped at **4**.

### D3 — Deprecation hygiene (v3 → v4 → v5 transitions)

*When a contract version is superseded, are old deploys archived, old scripts
gated, old tests either kept passing or removed, broadcast records pruned of
implicit-current status, UI repointed in the same PR?*

| Score | Anchor |
|------|--------|
| **0** | Multiple versions of the same logical contract are "live" with no version word visible to a reader; UI silently uses one, docs another, daemon a third; old deploy scripts run by default without warning. |
| **3** | Version words ("v3", "v4", "v5") appear across surfaces but disagree (e.g. docs page heading "v3" while JS uses a "v5" address); old `broadcast/` runs are not separated from the active version; old deploy scripts (`Deploy*V3.s.sol`) live alongside the current one with no `@deprecated` marker. |
| **5** | The active version is consistently identified in UI + docs + headline daemon configs; old broadcast records exist but are *not* repointed in the operator scripts; a `legacy/` or `archive/` convention is *partially* applied. |
| **7** | Superseded deploys are moved to `deployments/legacy/` (or annotated `deprecated: true` + `replaced_by: <addr>` in the manifest); old deploy scripts moved to `script/legacy/` or annotated `@deprecated`; UI hides or labels deprecated instances; PR description for every cutover lists every address-change call site. |
| **9** | All of (7), plus the address-set diff (§3) returns the empty set across `site*/`, `docs/`, `script/`, `script/agents/`, and `broadcast/active/`; no `.todo` / `.todo-v3` test files exist; daemons throw on startup if an env-var address belongs to a known-deprecated registry; Topic-2 e2e covers at least one legacy-instance read path. |

Dependency cap: if `topic-2` < 5.0 → D3 capped at **6** (no e2e coverage ⇒ no
proof that old or new paths still work after cutover).

### D4 — Operator surface readiness (daemons, runbooks, monitoring, alerting)

*Are the off-chain daemons resilient, observable, alertable, and runnable from
a one-page runbook?*

| Score | Anchor |
|------|--------|
| **0** | No daemons. Operator manually pokes contracts via `cast`. |
| **3** | Daemons exist as shell or Python scripts (`script/agents/auto_promote.sh`, `collect_metrics.py`, `script/daemon/submit.py`) but have inconsistent env-var defaults between header docs and body code; no heartbeat; no `--dry-run`; no documented restart contract. |
| **5** | Each daemon has a single config block, `--dry-run`, non-zero exit on unhandled failure, idempotent reads; header-vs-body env-var defaults agree; a short `docs/operations/` README describes the run command for each daemon. |
| **7** | All of (5), plus a heartbeat metric (last successful loop ts) written to a file *and* an HTTP/Prometheus endpoint; documented thresholds for "stale heartbeat", "balance low", "queue head not advancing"; a one-page runbook per recurring task (`deploy-instance`, `rotate-operator-key`, `pause-sale`) each with a test plan. |
| **9** | All of (7), plus an external scrape (Grafana Cloud / Prometheus / cron-heartbeat file) is wired and alert routes documented; on-chain `lastSeen` events let any third party verify daemon liveness without operator ingress; operator key rotation is a `cast`-driven runbook that has been exercised at least once; failure axes are mirrored in chaos tests under Topic-2 / Topic-4. |

### D5 — Maintainability and freshness (docs ↔ code; dead code; CI gating)

*Are docs date-fresh and address-correct? Is dead / orphan code culled? Is CI
gating the seams (not just unit tests)?*

| Score | Anchor |
|------|--------|
| **0** | Docs reference renamed files; addresses in docs do not match the addresses in the running UI; `.todo` / `.todo-v3` test files litter the tree; CI runs nothing seam-relevant. |
| **3** | Docs are mostly readable but ≥ 1 file path in markdown does not resolve on disk (e.g. `site-testnet/README.md` mentions a `registry.js` that was renamed to `shared.js`); CI runs `forge build` + `forge test` only — no UI test, no link-check, no manifest-vs-broadcast diff, no docs-staleness check. |
| **5** | Docs paths all resolve; address tables in docs match the manifest; a "last verified" date appears on critical address tables; CI runs format + tests + at least one cross-layer check (e.g. ABI lint, manifest existence). |
| **7** | All of (5), plus generated docs wherever possible (forge-doc, TypeDoc); `audit/` evaluators run on PRs and fail when topic-2/3/4 scores regress; address-set diff is a CI step; no orphan deploy scripts (every `script/*.s.sol` is referenced by an active runbook or annotated `@deprecated`). |
| **9** | All of (7), plus a CI job verifies that every backticked path in `*.md` exists on disk; every `0x[a-fA-F0-9]{40}` in the tree is matched against the manifest; every UI hardcoded ABI is replaced by a generated binding; mutation/coverage gates (Topic 4) and e2e gates (Topic 2) both block merges. |

Dependency cap: if `topic-1` < 5.0 → D5 capped at **6** (a beautiful UI on
stale contracts is still drift; but conversely, a fresh-but-unusable UI doesn't
deserve a high holistic maintainability score).

### D6 — Supply-chain risk (npm pin, foundry submodule pin, RPC, CDN, verification)

*Are external dependencies pinned by SHA / tag, are CDN scripts SRI-hashed, is
Etherscan verification complete, is the third-party hosting documented?*

| Score | Anchor |
|------|--------|
| **0** | Frontend loads `ethers` / `viem` from a CDN without integrity hash; `.gitmodules` pins are floating; vendored libs (reality-eth, seer, sx-evm, conditional-tokens) have no `VENDORED.md` recording upstream + commit + reason; zero contracts verified on Etherscan. |
| **3** | Some pins exist; CDN script tags lack `integrity="sha384-..."`; ≥ 1 vendored lib is undocumented; only the most prominent contracts are Etherscan-verified; Cloudflare Pages / Vercel host is not documented anywhere. |
| **5** | All `.gitmodules` are commit-pinned; OZ Contracts pinned to a named tag; CDN script tags carry SRI hashes; the *operator-touchable* contracts are Etherscan-verified; vendored libs each have a `VENDORED.md` or equivalent header comment; deployment host (Cloudflare Pages) is documented. |
| **7** | All of (5), plus a documented CVE-watch process for OZ / Foundry / Solidity; SRI hashes for *every* third-party CDN asset (no `https://cdn.jsdelivr.net/...` without integrity); **every** deployed contract on the live network is verified on Etherscan (including registry / sale / orchestrator / resolver / adapter); RPC endpoints are either self-hosted or use a documented provider with failover. |
| **9** | All of (7), plus dependency upgrades are PR-reviewed (no Dependabot auto-merge); a public attestation page maps each on-chain bytecode hash to its source-tag; the deploy pipeline produces reproducible bytecode (`forge build --use=<exact_solc>` + locked `via_ir` flag); a `SECURITY.md` exists with a coordinated-disclosure policy. |

## 2. Dependency caps (the meta layer)

Topic 5 is a meta-rubric. The evaluator MUST read the latest evaluations of
topics 1–4 and apply the following caps to raw scores:

| Cap | Trigger | Rationale |
|-----|---------|-----------|
| D2 ≤ 6 | `topic-3` < 5.0 | No spec ⇒ no checkable contract-of-contract for upgrades. |
| D2 ≤ 6 | `topic-4` < 5.0 | No fuzz / invariants ⇒ no defence-in-depth proof. |
| D2 ≤ 4 | `topic-3` < 3.0 OR `topic-4` < 3.0 | Combined absence of spec + tests blocks high security claims. |
| D3 ≤ 6 | `topic-2` < 5.0 | No e2e ⇒ no proof old/new paths work after a cutover. |
| D5 ≤ 6 | `topic-1` < 5.0 | UI-coupled freshness is meaningless when UI is itself weak. |

If multiple caps apply to the same dimension, the **tightest** cap wins.

The evaluator emits, alongside scores, a line per cap applied (or "no caps
applied" if all topics 1–4 score ≥ 5.0). This makes the meta dependency
auditable from a single output blob.

## 3. Entropy probes (cheap deterministic checks)

These are required CI / evaluator commands; they directly feed D1, D3, D5, D6.

| Probe | Command | Feeds |
|-------|---------|-------|
| Address-set per directory | `grep -ohE '0x[a-fA-F0-9]{40}' site-testnet/ docs/ script/ broadcast/ | sort -u` | D1, D3 |
| Version-word histogram | `grep -ohEr '\bv[0-9]\b' src/ docs/ site-testnet/ script/ | sort | uniq -c` | D3 |
| Stale path references | for f in `*.md`: extract backticked paths; `test -e` each | D5 |
| Orphan / `.todo` files | `find . -name '*.todo*' -not -path './lib/*'` | D3, D5 |
| Header-vs-code env defaults | Parse leading `#` block + `: "${VAR:=...}"` defaults; compare | D4 |
| Inline ABI literal count | `grep -cE "const +[A-Z_]+_ABI *= *\[" site-testnet/*.js` | D1 |
| CDN integrity audit | `grep -E '<script[^>]+cdn[^>]+>' site-testnet/*.html | grep -v integrity` | D6 |
| Etherscan-verified coverage | For each address in manifest, query Etherscan API for `getsourcecode` | D6 |
| Daemon header drift | Diff env-var defaults in header comment vs body of `script/agents/*.sh` | D4 |

Each probe must run in < 10s on a cold checkout; the cumulative budget for
all probes is < 60s so the CAO loop is not gated by topic 5's checks.

## 5. Stateless evaluator runbook

The evaluator should follow this script verbatim per scoring pass:

1. Read the most recent `audit/evaluations/topic-{1,2,3,4}-*.{json,md}`. Extract
   each topic's aggregate score. If a file is missing, assume 5.0 and log the
   assumption.
2. Run the entropy probes from §3. Capture each probe's output as evidence
   tied to a specific dimension.
3. For D1–D6, pick the anchor whose description is fully satisfied by §4's
   evidence (plus any new probe output); interpolate at even integers.
4. Apply caps from §2. The final score is `min(raw, cap)`.
5. Emit the report from §0 plus a 50-word delta versus the prior pass under
   `audit/evaluations/topic-5-*.json` (if any).

The evaluator must NOT:
- Re-derive Topic 1–4 scores from the source tree (the upstream rubrics own
  that work).
- Penalise the absence of mainnet deployment when the repo is testnet-only —
  but DO penalise pretending testnet config is production-safe.
- Reward documentation that exists but contradicts the running code; in such
  cases the doc *worsens* the score relative to no doc.
- Score dimensions above their cap, even if the raw evidence would justify it.

---

## 6. Regression rules

A subsequent pass is considered a **regression** (and the loop must surface
this to a human) if any of the following hold:

- Any dimension's `final_score` decreases by ≥ 1.0 without a documented
  rationale in the PR description.
- A new hardcoded address appears in `site-testnet/` that does not also exist
  in the manifest.
- A new `.todo` / `.todo-v3` file appears under `test/`, `script/`, or
  `site-testnet/`.
- The address-set probe surfaces a *new* divergence between any two of
  `site*/`, `docs/`, `script/`.
- The header-vs-code env-var drift probe surfaces a new mismatch.
- A previously Etherscan-verified contract reverts to unverified (typically
  after a redeploy without re-verification).
- `topic-1`, `topic-2`, `topic-3`, or `topic-4` aggregate regresses while the
  topic-5 raw score does not — implies dependency caps are not being applied.

A pass that fixes ≥ 1 regression item without introducing a new one is
**progress**, even if the aggregate score is unchanged.
